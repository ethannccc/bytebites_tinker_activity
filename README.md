Summary:

Students needed to understand how object-oriented design maps requirements into clear class responsibilities, especially how User, Food, FoodCollection, and Transaction interact without becoming tightly coupled. The hardest part is usually translating UML/scaffold ideas into method behavior with the right validation rules, return types, and state constraints. Another common struggle is balancing abstraction. Students either overbuild with extra classes or underbuild by skipping important invariants. AI was helpful for quickly generating scaffolds, suggesting method signatures, and producing test cases that cover core workflows and edge cases. AI was most misleading when it confidently claimed edits or execution status without actually verifying the current file state or when it introduced assumptions not explicitly in the spec. To guide a student without giving away the answer, I’d ask them to pick one user story and walk backward: what data must each class own, what method enforces each rule, and what test would prove it works. This keeps them focused on reasoning from requirements rather than memorizing code patterns.
